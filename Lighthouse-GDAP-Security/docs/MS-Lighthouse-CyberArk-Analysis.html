Analysis

CyberArk vs. Microsoft Lighthouse/PIM

Where Each Fits, Why CyberArk Cannot Replace PIM for GDAP, and How to Combine Them
Prepared 2026-04-07 | For MSP Security Architects with Existing CyberArk Deployments

Executive Summary

The Bottom Line

CyberArk cannot replace Entra PIM for GDAP access control. PIM for Groups is the only mechanism that can grant or revoke GDAP security group membership -- no external tool can do this. However, CyberArk fills critical gaps that PIM does not address: session recording, credential vaulting/rotation, non-Microsoft system access, and forensic audit trails. The two tools are complementary, not competing. The optimal architecture uses both in a layered model.

This document explains exactly why CyberArk cannot fully replace PIM, where each tool excels, the pros and cons of each approach, and how to combine them to maximize security without unnecessary friction or license costs.

1. Why CyberArk Cannot Replace PIM for GDAP

1a. The Architectural Constraint

GDAP access flows through a specific three-stage mechanism in Microsoft's infrastructure:

GDAP Access Chain:

Partner Tenant Security Group --[membership]--> GDAP Relationship --[role delegation]--> Customer Tenant Entra Roles

The critical link is security group membership. A technician accesses a customer tenant because they are a member of a specific security group that has been assigned to a GDAP relationship. The only system that can control membership in an Entra ID security group is Entra ID itself -- specifically, Entra PIM for Groups.

CyberArk operates at the credential layer (vaulting passwords, injecting credentials into sessions, recording sessions). It does not have the ability to add or remove users from Entra ID security groups as part of a JIT workflow. CyberArk's API can call Microsoft Graph to manage group membership in theory, but this would be a custom integration, not a supported native workflow, and it would still be calling Entra's APIs under the hood.

The Hard Technical Truth

No external PAM tool -- CyberArk, BeyondTrust, Delinea, or any other -- can natively manage GDAP security group membership. This is a Microsoft platform constraint. The JIT activation for GDAP must go through Entra PIM for Groups. This is not a preference or a recommendation; it is an architectural requirement of how GDAP works.

1b. What CyberArk CAN Do for Microsoft 365 Access

CyberArk is not useless for Microsoft 365 -- far from it. But it operates at a different layer:

3. Pros and Cons

Entra PIM + Lighthouse -- Pros

  • Only option for GDAP JIT -- no alternative exists
  • Zero infrastructure -- fully SaaS, no connectors or VMs
  • Native integration -- built into the same platform as GDAP, CA, Entra ID
  • Lower cost -- Entra P2 only in the partner tenant (~$9/user/month)
  • Lighthouse wizard -- simplified setup for GDAP templates with JIT
  • Customer visibility -- PIM activations visible in customer audit logs
  • Single identity plane -- no separate system to manage or break

Entra PIM + Lighthouse -- Cons

  • No session recording -- logs THAT a role activated, not WHAT user did
  • No credential vaulting -- passwords visible to users if they know them
  • No password rotation -- must be handled separately
  • M365/Azure scope only -- cannot manage Linux, DB, network device access
  • Template-wide JIT blast radius -- activates across all tenants in template
  • No ServiceNow native integration -- requires Power Automate for ITSM
  • Microsoft lock-in -- entire PAM posture depends on one vendor

CyberArk -- Pros

  • Session recording -- forensic-quality screen/keystroke capture for compliance
  • Credential vaulting + rotation -- passwords never exposed to technicians
  • Cross-platform -- Windows, Linux, databases, network devices, cloud
  • Zero Standing Privilege (SIA) -- ephemeral accounts eliminated
  • ServiceNow integration -- native approval workflow integration
  • MSP Hub -- centralized multi-tenant management console
  • Vendor-independent -- works across Microsoft, AWS, GCP, on-prem
  • EPM -- endpoint privilege management that PIM cannot touch

CyberArk -- Cons

  • Cannot manage GDAP group membership -- fundamental architectural gap
  • Higher cost -- CyberArk licensing + infrastructure + expertise
  • Infrastructure required -- even Privilege Cloud needs connector VMs
  • Complexity -- significant deployment and operational expertise needed
  • No native PIM integration -- separate approval and audit workflows
  • Potential user friction -- PSM proxy adds latency; some workflows feel different
  • RDS licensing -- self-hosted PSM requires Microsoft RDS CALs (eliminated in Privilege Cloud)

4b. The "Both" Scenario: PIM + PSM for High-Sensitivity M365 Access

For the most sensitive roles (eDiscovery Admin, Compliance Data Admin, Global Admin), the hybrid approach uses both tools simultaneously:

1

Technician activates PIM group membership (GDAP role becomes active in customer tenant)

2

Technician launches browser session through CyberArk PSM (session isolation + recording begins)

3

Technician performs admin work in customer tenant -- all actions recorded by PSM

4

PIM membership auto-expires (GDAP access revoked automatically)

Audit trail: PIM logs capture role activation/deactivation. CyberArk PSM captures the full session recording. Entra audit logs capture the specific admin actions. Three layers of evidence.

5. Decision Framework: When to Use What

PIM

Scenario: Pure M365/Azure MSP (no on-prem, no non-Microsoft systems)

Use PIM + Lighthouse only. CyberArk adds cost and complexity without addressing a gap that matters for your environment. Entra P2 in the partner tenant is sufficient. Consider CyberArk only if you need session recording for compliance (SOC 2 "what did they do" requirement).

Both

Scenario: MSP with CyberArk already deployed for infrastructure PAM

Use both in the hybrid model. PIM handles GDAP access (it has to). CyberArk handles everything it already handles (servers, databases, network devices). Add PSM session recording for high-sensitivity M365 roles if compliance requires it. No new CyberArk licensing needed if you already have Privilege Cloud -- just extend configuration.

Both

Scenario: MSP in regulated industry (HIPAA, SOX, PCI, CMMC)

Use both -- session recording is likely required. PIM provides the JIT access control. CyberArk PSM provides the session recording that auditors want. The "what did the admin do" question cannot be answered by PIM alone. Budget for CyberArk Privilege Cloud SaaS to minimize infrastructure overhead.

CA

Scenario: MSP evaluating CyberArk as a replacement for all PAM

CyberArk cannot replace PIM for GDAP. You still need Entra P2 and PIM for Groups. CyberArk can be the primary PAM for everything else, but the GDAP layer must go through PIM. Plan your architecture and licensing accordingly -- do not assume CyberArk eliminates the Entra P2 requirement.

7. Integration Points (Connecting the Two Systems)

No Native PIM-CyberArk Integration Exists

There is currently no out-of-the-box integration between Entra PIM approval workflows and CyberArk approval workflows. They operate as independent systems with separate audit trails. Integration requires custom work via ServiceNow or Power Automate.

7a. ServiceNow as the Common Broker

For MSPs using ServiceNow as their ITSM platform, it can serve as a unifying layer:

IntegrationHowResult
CyberArk --> ServiceNow Native CyberArk integration -- credential checkout requires a ServiceNow ticket All CyberArk access tied to trackable tickets
PIM --> ServiceNow Power Automate triggered on PIM activation events creates corresponding ServiceNow records All PIM activations tied to trackable tickets
ServiceNow = single source of truth Both CyberArk and PIM events flow to ServiceNow Unified audit trail for all privileged access, regardless of tool

7b. Sentinel as the Unified Monitoring Layer

Microsoft Sentinel can ingest audit data from both systems:

SourceWhat Flows to SentinelHow
Entra PIM Role activations, approvals, expirations Entra ID data connector (native)
CyberArk Privilege Cloud Credential checkouts, session starts/stops, policy violations CyberArk Sentinel data connector or Syslog/CEF forwarding
Correlation Cross-reference PIM activations with CyberArk sessions KQL join on timestamp + user identity

Final Verdict
PIM is required. CyberArk is complementary. Use both where the risk and compliance requirements justify the investment. Do not over-engineer: 85% of daily MSP work needs only PIM.
