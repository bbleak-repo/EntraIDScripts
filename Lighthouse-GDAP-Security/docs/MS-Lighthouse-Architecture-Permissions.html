1. GDAP Architecture Overview

Granular Delegated Admin Privileges (GDAP) replaced legacy DAP to provide per-customer, time-bound, role-specific delegated access for Cloud Solution Provider partners.

Three-Stage Permission Flow

1

Partner Creates Relationship Request

Specifies relationship name, requested Entra ID built-in roles, and duration (1-730 days). Generates a unique, single-use approval link for the customer.

2

Customer Approves via Unique Link

Customer Global Admin opens link in M365 Admin Center. Mandatory 15-30 second pause before the "Next" button becomes active. Customer sees all requested roles before consenting.

3

Partner Assigns Security Groups

MSP admin assigns partner tenant security groups to the approved relationship in Partner Center. This step connects approved roles to actual technicians. Without it, no one has access.

CRITICAL: Approval != Access

Many MSPs stop after Stage 2 and wonder why technicians cannot access customer tenants. Security group assignment (Stage 3) is required. Lighthouse surfaces a warning for relationships with no group assignments, but it is easily missed at scale.

2. GDAP Service Principals

When a GDAP relationship is established, three Microsoft-managed first-party service principals are automatically installed in the customer tenant. These enable the cross-tenant delegation mechanism.

Service PrincipalFunction
Partner Customer Delegated Admin Offline ProcessorProcesses GDAP relationship lifecycle events asynchronously
Partner Customer Delegated AdminHandles real-time cross-tenant authentication and role resolution for GDAP sessions
Partner Customer Delegated Admin Native AppNative application component supporting partner portal access to customer resources

These service principals are managed by Microsoft and cannot be modified. They are visible in the customer tenant's Enterprise Applications blade under "Microsoft Applications." The XTAP (Cross-Tenant Access Policy) "isServiceProvider" flag is set on the partner organization entry.

3. Permission Tiering Model

Roles should be organized into tiers with proportional security friction. Higher-impact roles require more gates before activation.

Tier 1 -- Help Desk (Low Risk)

PIM: Auto-activation, no approval Duration: 8-10 hours Use: Daily operations

Technician provides justification/ticket number and completes MFA. Membership activates automatically. Handles the majority of daily MSP work.

Global Reader Helpdesk Administrator License Administrator Service Support Administrator User Administrator Password Administrator

Tier 2 -- Specialist (Medium Risk)

PIM: Activation + ticket justification + MFA Duration: 1-2 hours Use: Workload-specific admin

Requires explicit ticket reference and MFA re-authentication. Short duration forces tightly scoped work sessions. Re-activate if more time needed.

Exchange Administrator Intune Administrator Security Administrator Security Reader SharePoint Administrator Teams Administrator Authentication Administrator

Tier 3 -- Escalation (High Risk, Approval Required)

PIM: Human approver + MFA + justification Duration: <1 hour Use: Rare escalations only

Requires approval from a designated security officer or management approver in addition to MFA and justification. These roles are touched rarely.

Privileged Authentication Admin Privileged Role Administrator Conditional Access Administrator Application Administrator Compliance Administrator

Global Administrator -- Emergency Only

PIM: Named approver + MFA + justification Duration: 1-4 hours max GDAP: Separate 90-day relationship

Dedicated GDAP relationship with 90-day duration. Auto-extend is prohibited by Microsoft for Global Admin roles. Requires renewed customer consent at expiry. Never standing access.

Global Administrator

8. Key Architectural Constraints

Roles Cannot Be Modified After Creation

Once a GDAP relationship is approved, its role set is fixed. To add or remove roles, you must create an entirely new relationship and obtain fresh customer approval.

Global Admin Auto-Extend Prohibited

GDAP relationships that include the Global Administrator role cannot use the 180-day auto-extend feature. Microsoft intentionally forces periodic customer re-approval for this role.

Lighthouse JIT Activates Across All Template Tenants

When a technician activates JIT access through a Lighthouse-configured access package, the elevation applies to every customer tenant attached to that GDAP template simultaneously -- not per-customer. MSPs needing per-customer JIT scoping must use multiple templates.

PIM Requires Entra ID P2 in the Partner Tenant

PIM for Groups (the GDAP JIT mechanism) requires Entra ID P2 or Microsoft 365 E5 licensing in the MSP's own tenant. Customer tenants do not require P2 for this.

Customer Sees Redacted Identity

Customer audit logs display MSP technicians as "[Partner Name] Technician" -- not individual names. Forensic correlation to a specific person requires MSP cooperation and cross-referencing with the partner tenant's sign-in logs.

Azure RBAC Via GDAP Derives From Admin Agents

Azure portal access in GDAP is not granted by specific GDAP role permissions. Instead, it is derived from the partner security group being nested under the Admin Agents group. Any Entra ID role (even Directory Readers) grants Azure subscription owner access when this nesting exists.
