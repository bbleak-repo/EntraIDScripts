1. Conditional Access Controls

1a. Cross-Tenant Access Settings

The customer tenant controls what trust is extended to the MSP partner tenant through three inbound trust checkboxes.

Trust SettingEffect When EnabledSecurity Implication
Trust MFA Accept MFA claims from the partner's home tenant MSP technician does not re-authenticate in customer tenant
Trust Compliant Devices Accept Intune device compliance from partner tenant Customer trusts MSP's Intune compliance policies
Trust Hybrid-Joined Devices Accept Entra hybrid join status from partner tenant Customer trusts MSP's on-prem AD + device management
CRITICAL: GDAP MFA Trust Override

For GDAP relationships (service provider access type), MFA from the partner's home tenant is ALWAYS trusted in the customer tenant -- regardless of whether the "Trust MFA" checkbox is enabled. This is a hardcoded behavior in the GDAP authentication flow. Customers cannot override it. This means the MSP's MFA security posture directly becomes the customer's MFA security posture for delegated access.

1b. US-Only Geo-Fencing

Restricting MSP access to US-based IP addresses only. Target CA policy to the "Service provider" user type scoped to the MSP's tenant ID.

Create Named Location -- In Entra ID > Conditional Access > Named Locations, create "United States" using IP-based geolocation (Microsoft's country-to-IP mapping).

Create CA Policy -- Target: "Service provider" user type. Scope to the MSP's specific tenant ID. Apply to "All cloud apps."

Set Condition -- Locations > Include: "Any location." Exclude: "United States" named location.

Set Grant -- "Block access." This blocks all MSP access from non-US IP addresses.

WARNING: VPN/Proxy Bypass Risk

IP-based geo-fencing can be bypassed if the MSP technician uses a US-based VPN or proxy. The customer's CA policy sees the VPN exit IP, not the user's physical location. Mitigation: Deploy Microsoft Global Secure Access with source IP restoration to use the original client IP rather than the proxy IP.

1e. Device Compliance for Partner Access

Device Trust Requires Explicit Opt-In

MSP devices are enrolled in the MSP's Intune instance, not the customer's. Without the "Trust compliant devices" checkbox enabled in inbound cross-tenant access settings, any CA policy requiring device compliance will block ALL MSP technicians. The customer must explicitly choose to trust the MSP's device compliance claims.

2. Privileged Access Management / JIT

2a. Entra PIM for GDAP (Native JIT)

PIM for Groups = The ONLY Mechanism for GDAP JIT

No external tool -- including CyberArk -- can grant or revoke GDAP security group membership. Entra PIM is the native and only production-grade option for controlling just-in-time access to GDAP roles. CyberArk cannot replace it for this specific function.

2c. Hybrid PAM Architecture

Entra PIM Handles

  • GDAP security group JIT activation
  • Role eligibility and time-bound activation
  • Approval workflows for high-privilege roles
  • Activation audit logging
  • Microsoft 365 / Entra ID scope

CyberArk Handles

  • Credential vaulting and rotation
  • Full session recording (screen, keystrokes, commands)
  • Non-Microsoft system access (Linux, network, DB)
  • Break-glass credential management
  • PSM for forensic audit compliance
No Native Integration

There is no native integration between Entra PIM approval workflows and CyberArk approval workflows. They run as parallel systems with separate audit trails. Organizations operating both must maintain two approval and review processes.

2d. Session Recording Gap

PIM Records "THAT" -- Not "WHAT"

PIM logs that a role was activated (who, when, which role, justification). It does NOT record what the user did after activation. CyberArk PSM provides forensic session recording (screen capture, keystroke logging, command recording) required for SOC 2, ISO 27001, and NIST audit compliance. Organizations requiring full session forensics must implement CyberArk PSM or equivalent alongside PIM.

3b. Sentinel KQL Queries

KQL: Service Provider Sign-in Filtering
Filter Entra sign-in logs for MSP partner access
SigninLogs | where CrossTenantAccessType == "serviceProvider" | project TimeGenerated, UserDisplayName, UserPrincipalName, IPAddress, Location, HomeTenantId, ResourceTenantId, AppDisplayName, Status | sort by TimeGenerated desc
KQL: Privileged Role Assignment Detection
Detect when GDAP partners assign or activate privileged roles
AuditLogs | where OperationName has_any ("Add member to role", "Add eligible member to role") | where InitiatedBy.user.userPrincipalName has "#EXT#" or InitiatedBy.user.userPrincipalName == "" | extend TargetRole = tostring(TargetResources[0].displayName) | project TimeGenerated, OperationName, TargetRole, InitiatedBy, Result | sort by TimeGenerated desc
KQL: Azure Lighthouse Delegation Changes
Monitor additions and deletions of Lighthouse registration assignments
AzureActivity | where OperationNameValue =~ "Microsoft.ManagedServices/registrationAssignments/write" or OperationNameValue =~ "Microsoft.ManagedServices/registrationAssignments/delete" | project TimeGenerated, Caller, CallerIpAddress, OperationNameValue, ActivityStatusValue, ResourceGroup, SubscriptionId | sort by TimeGenerated desc

union workspace("customer-a-workspace").SigninLogs, workspace("customer-b-workspace").SigninLogs, workspace("customer-c-workspace").SigninLogs | where CrossTenantAccessType == "serviceProvider" | where ResultType != "0" // Failed sign-ins | summarize FailedAttempts = count() by UserPrincipalName, IPAddress, bin(TimeGenerated, 1h) | where FailedAttempts > 5

3d. Critical Audit Gaps

Purview Interactive Search NOT Supported via GDAP

The "Search audit log" operation in the Purview compliance portal does not work for GDAP-authenticated users because the identity does not exist as a user object in the customer tenant. Workarounds: PowerShell (Search-UnifiedAuditLog), Office 365 Management Activity API, or a dedicated user account in the customer directory.

MSP Technician Identity Redacted

Customer-facing logs display MSP technicians as "[Partner Name] Technician" -- individual names are privacy-redacted. The user's object ID is present in the log record, but correlating it to a specific person requires the MSP to look up the object ID in their own tenant. Forensic investigations require MSP cooperation.

Preview: Native GDAP Sentinel Access (April 2026)

A new consent-based three-way handshake model is entering preview that allows GDAP-based Sentinel access directly through the Defender portal, as an alternative to the current Azure Lighthouse delegation model for Sentinel workspace access.

4. PRT & Device Trust

4a. PRT Authentication Flow for MSP Access

Windows Login -- MSP technician logs into their device. CloudAP plugin issues PRT request to partner tenant's Entra ID. PRT is bound to the device's TPM 2.0 chip (session key never leaves hardware).

Lighthouse Portal Access -- Technician navigates to lighthouse.microsoft.com. WAM broker intercepts the auth request and presents PRT + TPM-signed proof-of-possession.

Cross-Tenant Auth -- Entra ID validates PRT claims: device ID, MFA claim, device compliance, location. Issues access token scoped to the customer tenant via GDAP relationship.

Customer CA Evaluation -- Customer tenant's Conditional Access evaluates the token. Trust settings determine which partner claims are honored (device compliance, MFA). GDAP MFA is always trusted.

Access Granted -- Technician accesses customer resources with the Entra roles assigned to their GDAP security group. Session subject to customer's session lifetime controls.

4b. PRT Security Risks (MSP-Specific)

Pass-the-PRT Attack

If a PRT is extracted from an MSP technician's device, the attacker gains SSO to ALL customer tenants the technician has GDAP access to. The blast radius is the entire MSP portfolio, not a single customer.
