Best Practices

Securing Microsoft Lighthouse for MSP Partners

SAML/OIDC Integrations | Sentinel | Purview | Defender | Security Hardening
Prepared 2026-04-07 | For MSP Security Architects Planning Azure Service Integrations

Document Purpose

This guide covers best practices for an MSP partner planning to secure their Lighthouse/GDAP deployment while integrating Azure SAML/OIDC-based applications, standing up Microsoft Sentinel for multi-tenant monitoring, configuring Purview for compliance visibility, and deploying Defender services across customer tenants. Each section addresses the security implications specific to the MSP delegated access model.

Scope

This document assumes GDAP is already established (DAP fully deprecated as of July 2024). It focuses on securing the operational environment around Lighthouse, not the initial GDAP setup. For GDAP setup SOPs, see the companion SOP template.

1. Partner Tenant Hardening (Foundation)

The MSP's own Entra ID tenant is the single most critical security boundary. Every customer tenant is reachable through it. Harden it first.

1a. Identity Security Baseline

  • Phishing-resistant MFA for all admin accounts -- FIDO2 security keys or Windows Hello for Business. Microsoft Authenticator push is not phishing-resistant. GDAP mandates MFA in the home tenant, and that MFA claim is always trusted by customer tenants (hardcoded -- customers cannot override this).
  • Separate admin and productivity accounts -- Technicians should never use their daily email account for GDAP access. Create dedicated adm-*@partner.com accounts for administrative work.
  • PIM for Groups on ALL GDAP security groups -- Convert all standing membership to eligible-only. No technician should hold active GDAP group membership by default. Requires Entra ID P2 in the partner tenant.
  • Emergency access (break-glass) accounts -- At least 2, excluded from Conditional Access, stored in a monitored safe with alerting on sign-in. Rotate credentials quarterly.
  • Conditional Access baseline in the partner tenant -- Block legacy authentication, require compliant devices for admin portals, enforce sign-in risk policies (Entra ID Protection P2).
  • Disable self-service password reset for admin accounts -- Prevent social engineering password resets on accounts with GDAP access.
  • Token protection (preview) -- Enable for admin accounts when GA. Binds tokens to the issuing device, preventing token replay from a different machine.

2. Azure SAML/OIDC Integration Security

When the MSP partner sets up SAML or OIDC-based application integrations in customer tenants via GDAP, several security considerations apply.

2a. App Registration and Enterprise Application Governance

GDAP Roles Required

TaskMinimum RoleTier
Register new applicationsApplication AdministratorTier 3 -- JIT
Configure SSO (SAML/OIDC)Application Administrator or Cloud Application AdministratorTier 3 -- JIT
Grant admin consent for permissionsPrivileged Role Administrator (for app-level) or Cloud Application AdministratorTier 3 -- JIT
Assign users/groups to appsUser Administrator or Application AdministratorTier 2
View app configurations (read-only)Global ReaderTier 1
CRITICAL: Application Administrator is a Tier 3 Role

Application Administrator can create credentials for any application, including those with privileged Graph API permissions. This is effectively a privilege escalation path. Never assign this as standing access. Use PIM with human approver, max 1-hour activation, and mandatory justification referencing a change ticket.

2b. SAML Configuration Best Practices

  • Use the Entra ID Gallery app template when available -- Pre-configured claim mappings reduce misconfiguration risk. Gallery apps get Microsoft's SSO testing and documentation.
  • Configure SAML token signing with SHA-256 -- Default for new configurations. Verify the signing algorithm is not SHA-1 on legacy configurations.
  • Set token encryption if the SaaS vendor supports it -- Encrypts SAML assertions in transit. Requires the vendor to provide their public certificate.
  • Configure appropriate NameID format -- Use user.userprincipalname for most SaaS apps. Avoid user.mail if UPN and email differ (common in hybrid environments).
  • Require user assignment -- On the Enterprise Application properties, set "Assignment required?" to Yes. This prevents any authenticated user from accessing the app -- only explicitly assigned users/groups can SSO.
  • Scope Conditional Access per application -- Create CA policies targeting specific enterprise applications rather than "All cloud apps" when the app warrants additional controls (MFA step-up, device compliance, location restrictions).
  • Test with a pilot group before broad rollout -- Assign a small test group, verify claims, then expand.

2c. OIDC/OAuth Best Practices

  • Prefer OIDC over SAML for new integrations -- OIDC uses JWTs with shorter lifetimes, supports PKCE for public clients, and integrates better with modern SPA/mobile architectures.
  • Use certificate credentials for confidential clients, not client secrets -- Client secrets can be leaked; certificates provide stronger proof-of-possession. Rotate certificates before expiry (set calendar reminders).
  • Scope API permissions to minimum required -- Never grant Directory.ReadWrite.All when User.Read suffices. Audit existing permissions quarterly.
  • Admin consent workflow -- Enable the admin consent workflow in the customer tenant so end-users can request permissions rather than self-consenting to apps. This prevents OAuth consent phishing attacks.
  • Monitor app consent grants -- Use Sentinel or Purview audit logs to detect new admin consent grants. Alert on Consent to application events, especially for high-privilege Graph scopes like Mail.ReadWrite, Files.ReadWrite.All, RoleManagement.ReadWrite.Directory.
OAuth Consent Phishing -- MSP Attack Vector

Attackers create malicious app registrations that request broad permissions. If an admin in the customer tenant grants consent (or if the tenant allows user self-consent), the attacker gains persistent API access. MSPs should: (1) disable user consent for apps in customer tenants, (2) enable the admin consent workflow, (3) alert on new admin consent events in Sentinel.

2d. Cross-Tenant App Considerations

When the MSP deploys multi-tenant applications that need to access customer tenant data (e.g., monitoring dashboards, reporting tools):

  • Register the app as multi-tenant in the partner tenant -- Set "Supported account types" to "Accounts in any organizational directory."
  • Customer admin consent is required per tenant -- The customer must grant admin consent for the app's permissions in their tenant. Document this in your onboarding runbook.
  • Use application permissions (not delegated) for service-to-service -- Service accounts running background jobs should use certificate-based app authentication, not delegated user tokens.
  • Track app credential expiry -- Both client secrets and certificates expire. Set monitoring for expiry dates. An expired credential means broken integration with no warning to end users.
  • Document all apps deployed in each customer tenant -- Maintain an inventory. This is critical for offboarding: when a customer relationship ends, all MSP-deployed apps must be removed.

3. Microsoft Sentinel -- Multi-Tenant MSP Deployment

3a. Architecture Decision: Data-Per-Customer vs. Centralized

Data-Per-Customer Tenant (Recommended)

Each customer has their own Log Analytics workspace and Sentinel instance. MSP accesses via Azure Lighthouse delegation.

  • Customer owns their data and retains full sovereignty
  • Compliance boundaries are clean (data never leaves the customer tenant)
  • Customer can view their own Sentinel without MSP involvement
  • Offboarding is clean -- revoke Azure Lighthouse, data stays with customer
  • Cost attribution per customer is straightforward

Best for: Most MSP engagements, regulated industries, customers who want data ownership.

Centralized in MSP Tenant

MSP maintains a single Sentinel workspace and ingests data from all customers.

  • Simpler SOC operations (one workspace, one set of rules)
  • Cross-customer correlation is native (no cross-workspace queries)
  • Lower operational overhead for small MSPs
  • Customer has no direct access to their own data
  • Data sovereignty issues (customer data in MSP tenant)
  • Offboarding requires data extraction/deletion

Best for: Small MSPs with non-regulated SMB customers who want "fully managed" SIEM.

3b. Azure Lighthouse + Sentinel Setup

For the data-per-customer model, Azure Lighthouse provides the cross-tenant access to customer Sentinel workspaces.

1
Deploy Azure Lighthouse registration -- Create an ARM template that defines the MSP's managing tenant ID and role authorizations. Minimum roles: Microsoft Sentinel Reader (for monitoring) + Microsoft Sentinel Responder (for incident management) + Log Analytics Reader (for workspace queries). Use PIM-eligible assignments for Contributor-level roles.
2
Deploy the template to each customer subscription -- Customer must accept the delegation. Use the Azure portal "Service providers" blade or ARM template deployment. This is separate from GDAP -- Azure Lighthouse and GDAP are parallel mechanisms for Azure vs. M365 access.
3
Enable Sentinel data connectors in each customer workspace -- Requires both Azure Lighthouse delegation AND GDAP roles for M365 data connectors. Key connectors: Microsoft Entra ID (sign-in + audit logs), Microsoft 365 Defender, Microsoft Defender for Cloud, Azure Activity.
4
Deploy analytics rules from a central repository -- Use Azure Lighthouse to push standardized detection rules across all customer workspaces. Maintain rules in a Git repo and deploy via Azure DevOps or GitHub Actions.
5
Configure Lighthouse false-positive suppression -- Microsoft 365 Lighthouse's background Graph API polling generates events in customer Sentinel workspaces. Create automation rules to suppress these. Filter on the Lighthouse service principal's application ID in the sign-in logs.

3c. Critical Sentinel KQL Detections for GDAP Monitoring

Detect MSP Partner Sign-Ins (Service Provider Access Type)
Baseline: All GDAP partner authentication events
SigninLogs | where CrossTenantAccessType == "serviceProvider" | project TimeGenerated, UserDisplayName, UserPrincipalName, IPAddress, Location, HomeTenantId, AppDisplayName, Status, ConditionalAccessStatus | sort by TimeGenerated desc
Detect Partner Sign-Ins from Unexpected Locations
Alert when partner access comes from non-US locations (if geo-fenced)
SigninLogs | where CrossTenantAccessType == "serviceProvider" | where Location !has "US" | project TimeGenerated, UserDisplayName, IPAddress, Location, AppDisplayName, Status | sort by TimeGenerated desc
Detect Privileged Role Activation by Partner Users
Monitor PIM activations for GDAP security groups
AuditLogs | where OperationName has_any ("Add member to role", "Add eligible member to role", "Activate PIM role", "Add member to group") | where InitiatedBy.user.userPrincipalName has "#EXT#" or InitiatedBy.user.userPrincipalName == "" | extend TargetRole = tostring(TargetResources[0].displayName) | project TimeGenerated, OperationName, TargetRole, InitiatedBy, Result | sort by TimeGenerated desc
Detect New Admin Consent Grants (OAuth Phishing Detection)
Alert on new application consent -- critical for SAML/OIDC security
AuditLogs | where OperationName == "Consent to application" | extend AppName = tostring(TargetResources[0].displayName) | extend ConsentedPermissions = tostring(TargetResources[0].modifiedProperties) | project TimeGenerated, AppName, ConsentedPermissions, InitiatedBy.user.userPrincipalName, Result | sort by TimeGenerated desc
Cross-Workspace Hunting Across Customer Tenants (up to 20)
Unified threat hunting across multiple customer workspaces
union workspace("customer-a-workspace").SigninLogs, workspace("customer-b-workspace").SigninLogs, workspace("customer-c-workspace").SigninLogs | where CrossTenantAccessType == "serviceProvider" | where ResultType != "0" // Failed sign-ins only | summarize FailedAttempts = count() by UserPrincipalName, IPAddress, bin(TimeGenerated, 1h) | where FailedAttempts > 5

Purview Portal Audit Search Gap

The interactive "Search audit log" in the Purview compliance portal does NOT work via GDAP. The partner's identity does not exist as a user object in the customer tenant, which the portal requires. Workarounds: (1) Use PowerShell Search-UnifiedAuditLog, (2) Ingest audit events via the Office 365 Management Activity API into Sentinel, (3) Create a dedicated service account in the customer tenant for audit access.

4b. Purview Security Considerations for MSP Access

  • Content Explorer access = automatic HLA (High-Level Access) -- A user with Content Explorer Content Viewer can view the actual contents of files matched by DLP policies, sensitivity labels, and retention labels. This is organization-wide access to sensitive data. Treat with CyberArk PSM session recording or equivalent.
  • Audit Manager = HLA -- Can modify audit retention policies. A malicious actor with this role can shorten retention periods to cover their tracks. Monitor changes to audit retention settings.
  • eDiscovery Administrator (Tier 1) vs. Manager (Tier 3) -- eDiscovery Administrator has org-wide access to all cases and can search any mailbox. eDiscovery Manager is scoped to their own cases only. Use Manager where possible; reserve Administrator for escalation.
  • DLP policies can disrupt business -- A misconfigured DLP policy can block email flow, file sharing, or Teams messages. Require change tickets for DLP modifications. Test in simulation mode before enforcement.
  • Sensitivity labels propagate broadly -- A published label policy affects all users in scope. Mislabeled data protection policies can lock users out of their own files. Use pilot groups and gradual rollout.

Lighthouse Only Surfaces Limited Defender Alerts

Only Defender for Business, Defender Antivirus, and Entra ID Protection alerts appear in the Lighthouse portal. For MDO, MDI, MCAS, and Defender for Cloud alerts, you must use the Defender XDR portal (security.microsoft.com) or Sentinel. Plan your SOC workflow accordingly.

5b. Defender Deployment Best Practices

  • Use Lighthouse baselines for consistent Defender deployment -- Lighthouse provides security baselines that can be pushed to customer tenants. These include Defender policies, CA policies, and device compliance policies. Customize the default baseline for your MSP's standards.
  • Enable Defender for Endpoint via Intune -- Use Intune device onboarding profiles (available via GDAP with Intune Administrator role). This ensures consistent onboarding across customer device fleets.
  • Configure Attack Surface Reduction (ASR) rules -- Start in audit mode, monitor for false positives across customer workloads, then move to block mode. ASR rules are the most impactful Defender for Endpoint configuration for SMB customers.
  • Enable Safe Links and Safe Attachments (MDO) -- For customers with Exchange Online P1/P2 or M365 Business Premium. Deploy via security policies in the Defender portal. These protect against email-borne phishing and malware.
  • Configure MCAS for OAuth app governance -- Detect when users consent to malicious applications (OAuth consent phishing). Create policies to alert on new app registrations with high-privilege permissions.
  • Deploy Defender for Cloud on Azure subscriptions -- Via Azure Lighthouse. Enable Defender CSPM (Cloud Security Posture Management) for free tier, or Defender for Servers/Storage/SQL for workload protection. This requires the Azure Lighthouse delegation, not GDAP.

5c. Multi-Tenant Defender XDR (Preview)

Defender XDR Multi-Tenant Management

Microsoft has been building native multi-tenant management into the Defender XDR portal (security.microsoft.com). As of early 2026, GDAP-based access in the Defender portal is expanding via a consent-based model. This allows MSPs to manage Sentinel and Defender incidents across customer tenants from a single Defender portal session, as an alternative to the Azure Lighthouse model. Monitor Microsoft's announcements for GA status.

Customer Cross-Tenant Trust Prerequisites

For device compliance policies to work, the customer must enable "Trust compliant devices" in their cross-tenant access inbound settings. Without this, the customer's CA policy will block all MSP technicians because their devices are enrolled in the MSP's Intune, not the customer's. Similarly, "Trust multifactor authentication" enables MFA claim passthrough (though GDAP always trusts partner MFA regardless). These settings require Entra ID P1 in the customer tenant.

7. Operational Security Practices

7a. Change Management for Customer Tenant Modifications

  • All SAML/OIDC app registrations require a change ticket -- Document the app name, vendor, permissions requested, business justification, and approver. This creates an audit trail and prevents unauthorized app deployments.
  • DLP and sensitivity label changes require simulation mode first -- Never deploy enforcement-mode DLP to production without a simulation period. Broken DLP rules can halt business operations.
  • Sentinel analytics rule changes tracked in Git -- Use infrastructure-as-code for Sentinel rules. Deploy via CI/CD pipeline. This provides version control, peer review, and rollback capability.
  • Quarterly GDAP relationship review -- Review all active relationships. Verify role sets still match SOW. Remove relationships for departed customers. Check for relationships approaching expiry.
  • Annual penetration test of the partner tenant -- Scope should include: credential phishing simulation, PIM bypass attempts, GDAP blast radius assessment, and device compliance policy testing.
